# История разработки crabsort

**crabsort** — утилита на Rust для автоматической сортировки файлов по типам и поиска дубликатов, с собственным TUI (терминальным интерфейсом). Проект развивается **11 дней** (14–24 марта 2026), содержит **97 коммитов**, **17 исходных файлов** на ~**2840 строк** Rust-кода.

---

## Фаза 1: Ядро сортировки файлов (14 марта — 14 коммитов)

Первый день — бурная активность, заложен фундамент всего проекта.

1. **Начало** (`e83dbc0` – `a5d8764`) — чтение содержимого директории, первые шаги.
2. **Определение типов файлов** (`b380882` – `6524cd9`) — чтение MIME-типов, создание карты типов (file types map), маппинг типов в целевые директории, вынос в отдельный модуль `file_types.rs`.
3. **Перемещение файлов** (`575e104`) — файлы реально раскладываются по папкам.
4. **Dry run** (`9e8c274`) — безопасный режим "только показать, не двигать".
5. **Рефакторинг** (`9f1552d`, `d5a0945`) — уменьшение вложенности, чистка кода. Уже на первый день виден рефакторинг — хороший знак.

**Итог фазы:** рабочий CLI-инструмент, который читает директорию, определяет типы файлов по MIME и раскладывает их по папкам.

---

## Фаза 2: Полировка и расширение типов (15 марта — 8 коммитов)

1. **Улучшение CLI** (`5db08c2`) — лучший парсинг аргументов.
2. **Verbose-режим** (`347840a`) — подсчёт файлов, вывод подробной информации.
3. **Расширение базы типов** (`f5ace50` – `238d9cb`) — 5 коммитов подряд на расширение поддерживаемых типов файлов, добавлен fallback по расширению.

**Итог фазы:** инструмент стал "пользовательским" — с нормальным CLI, verbose-выводом и широким покрытием типов.

---

## Фаза 3: Поиск дубликатов (15 марта — 16 коммитов)

Самая крупная и технически насыщенная фаза. Видна последовательная, инкрементальная разработка:

1. **Базовый поиск** (`0b857d3` – `c35c573`) — рекурсивный обход дерева, группировка по размеру файлов.
2. **Хеширование** (`452d7d6` – `82292d2`) — сначала partial hash (первые байты), потом полный хеш для уточнения.
3. **Параллелизация** (`789f181`, `271e1dd`) — параллельное вычисление хешей (partial и full) — ускорение на многоядерных системах.
4. **Оптимизация алгоритма** (`d581fea` – `07858aa`):
   - Минимальный порог 100KB (мелкие файлы не дублируются)
   - Хеширование первых и последних 4KB (вместо только начала)
   - Переход на **blake3** — один из самых быстрых криптохешей
5. **Arc-обёртки** (`e9e95a6`) — `Arc` для shared ownership в многопоточном контексте.
6. **Оценка экономии** (`995e2e3`) — подсчёт, сколько места можно сэкономить.
7. **Параллельный обход дерева** (`ee8df15`) — большой коммит (183 ins / 103 del), файловое дерево обходится параллельно.

**Итог фазы:** полноценный дедупликатор с продуманной стратегией (size → partial hash → full hash), параллелизмом и оценкой экономии. Отдельный модуль `file_duplicates.rs`.

---

## Фаза 4: TUI — терминальный интерфейс (15–17 марта — 24 коммита, ветка `feature/tui`)

Самая длительная фаза, разработанная в отдельной ветке и потом смерженная.

1. **Фундамент** (`dc785ee` – `450c20f`) — raw mode терминала, собственный drawable buffer.
2. **Отрисовка** (`b977c65` – `af26252`) — рисование боксов, меню, заголовки — **собственный UI-фреймворк с нуля**, без ratatui или cursive.
3. **Интеграция сортировки** (`7122881` – `6b085f0`) — dry run в TUI, предпросмотр списка файлов, структура `FileTreeItem`.
4. **Виджетная система** (`61d2185` – `93f4e97`) — выделены виджеты `UIMenu`, `UIFileList` в `src/widgets/`, трейт `Widget`.
5. **Архитектура событий** (`ea2a1af` – `2fdd10b`) — `EventBus` для развязки виджетов, переход от замыканий к строковым событиям в `MenuItem`.
6. **Меню и навигация** (`9787a40` – `13652e5`) — меню сортировки, кнопка выхода.
7. **Прогресс-бар и дубликаты** (`e718e70` – `6386b03`) — глобальный прогресс-бар, запуск поиска дубликатов в отдельном потоке с асинхронным обновлением прогресса, отображение отчёта.
8. **Merge** (`878b5e5`) — ветка влита в master.

**Итог фазы:** полноценный TUI со своим рендерером, виджетной системой, event bus, прогресс-баром и интеграцией обеих фич (сортировка + дубликаты).

---

## Фаза 5: Визуальная полировка и новые виджеты (17 марта — 7 коммитов)

Сразу после мержа TUI — серия доработок интерфейса и извлечение логики в отдельные модули.

1. **Жёлтая тема** (`63a364a`) — визуальная стилизация.
2. **Отображение размеров** (`f30a4a6`) — показ размеров файлов и возможной экономии в UI.
3. **Константы действий** (`ce4bf5e`) — строковые константы для событий меню, чистка кода.
4. **Диалог ввода** (`8eccbd5`) — новый виджет `UIInputDialog` для пользовательского ввода.
5. **Ввод минимального размера** (`a9e420d`) — пользователь может указать порог размера дубликатов в килобайтах через диалог.
6. **Вынос логики сортировки** (`9de0c43`) — создан `file_sorting.rs` (~126 строк), `main.rs` разгружен.
7. **Прогресс-бар как виджет** (`b8dfc9c`) — вынос прогресс-бара в `UIProgressBar`, `main.rs` стал легче на ~80 строк.

**Итог фазы:** архитектура подчищена, логика распределена по модулям, появился диалог ввода — TUI стал интерактивнее.

---

## Фаза 6: Полноценная сортировка в TUI (18 марта — 9 коммитов)

Превращение сортировки из сухого "dry run" в реально работающую фичу с визуальным контролем.

1. **Кеширование rect** (`c953d6d`) — оптимизация отрисовки виджетов.
2. **Select + scroll** (`11c85f9`) — нормальная поддержка выбора и прокрутки в списке файлов.
3. **Динамический прогресс-бар** (`d411ae2` – `a62da27`) — заголовок и описание прогресс-бара обновляются по ходу работы.
4. **Динамическое описание дубликатов** (`ab88d93`) — прогресс поиска дубликатов показывает текущий этап.
5. **Работающая сортировка** (`198c950`) — файлы реально перемещаются из TUI, добавлен `Makefile`.
6. **Сортировка в потоке** (`c775873`) — сортировка запускается в отдельном потоке с прогресс-баром.
7. **Списки поддерживаемых/неподдерживаемых** (`cb68453`) — TUI показывает два списка: файлы, которые будут отсортированы, и файлы с неизвестными типами.

**Итог фазы:** сортировка полностью интегрирована в TUI — запуск в потоке, прогресс, разделение по категориям. Проект перестал быть "только для дубликатов".

## Фаза 7: Интерактивный выбор файлов (20–21 марта — 8 коммитов)

Ключевая фаза для UX — пользователь может выбирать конкретные файлы для операций.

1. **Значение по умолчанию** (`039bf10`) — `UIInputDialog` поддерживает default value.
2. **Реорганизация UI-кода** (`93d6a4a`) — чистка и перестройка UI-логики.
3. **Выделение файлов** (`28c0f2a`) — `UIFileList` поддерживает выбор элементов.
4. **Event bus в FileList** (`30fde29`) — `UIFileList` подключён к шине событий.
5. **Упрощение обобщений** (`f6c8074`) — замена `Fn` дженериков на `fn` указатели — упрощение типов.
6. **Реализация выбора** (`cf175fa` – `195b186`) — сначала черновая реализация, затем погрупповой выбор файлов.
7. **Select all** (`e4707a1`) — кнопка "выбрать все" для быстрого массового выбора.

**Итог фазы:** TUI стал по-настоящему интерактивным — пользователь может выбирать файлы, выделять группами, использовать "выбрать все".

---

## Фаза 8: Новые виджеты и стабилизация (22–24 марта — 8 коммитов)

Финальная на данный момент фаза — расширение набора виджетов и повышение надёжности.

1. **Группированный список** (`7019a9b`) — новый виджет `UIGroupedList` (214 строк) для отображения файлов по группам.
2. **Статус-бар** (`06ce9da`, `b04066c`) — новый виджет `UIStatusBar`, обработка состояния фокуса.
3. **Обработка resize** (`904e739`) — прогресс-бар и important-виджеты корректно реагируют на изменение размера терминала.
4. **Скрытие пустых списков** (`b289796`) — если нет неподдерживаемых файлов, список не показывается.
5. **Исправление параллельного обхода** (`8146b66`) — flatten параллельного сканирования директорий, добавлены safety-комментарии к `unwrap()`.
6. **Извлечение layout-методов** (`a87ecc6`) — повторяющиеся layout-вычисления вынесены в отдельные методы.
7. **Упрощение прогресс-бара** (`8d9be8d`) — убран `Arc<Mutex>` из прогресс-бара, введён `AppModal` enum для управления модальными состояниями приложения.

**Итог фазы:** UI обогащён группированным списком и статус-баром, код стабилизирован, убраны лишние блокировки, улучшена обработка resize.

## Ключевые наблюдения

- **Очень интенсивная разработка** — 97 коммитов за 9 активных дней, в среднем ~11 коммитов в день. Пиковые дни (14, 15, 17 марта) — до 24 коммитов.
- **Грамотная инкрементальность** — каждый коммит делает одну вещь, видна чёткая прогрессия.
- **TUI с нуля** — не использовались готовые TUI-фреймворки, всё написано руками (buffer, box, widgets, event bus).
- **Постоянный рефакторинг** — вынос логики в модули (`file_sorting.rs`, виджеты), упрощение типов (`Fn` → `fn`), удаление `Arc<Mutex>` где не нужно, извлечение layout-методов.
- **Оптимизация по ходу** — параллелизм, blake3, smart hashing (size → partial → full), rect caching, flatten parallel scan.
- **Эволюция виджетной системы** — от 2 виджетов (menu, filelist) до 6 (+ grouped list, input dialog, progress bar, status bar). Виджеты становятся самостоятельнее (event bus, кеширование rect).
- **От CLI к полноценному TUI-приложению** — проект эволюционировал от простого CLI-скрипта до интерактивного терминального приложения с модальными диалогами, групповым выбором файлов и статус-баром.
